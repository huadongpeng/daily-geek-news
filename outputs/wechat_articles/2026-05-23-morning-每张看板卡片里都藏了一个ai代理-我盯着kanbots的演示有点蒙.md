每张看板卡片里都藏了一个AI代理，我盯着Kanbots的演示有点蒙

正文内容
昨晚刷到 Kanbots 的演示视频，我端着杯速溶咖啡，盯着屏幕愣了好一会。

说真的，不是我没见过 AI Agent。这两年各种“自主代理”的论文和框架我追得不少，AutoGPT、CrewAI、MetaGPT 都 clone 到本地跑过 demo。但它们给我的感觉一直是“实验室里的东西”——得配环境、写配置、处理各种报错，搞完一圈觉得累，又默默关掉回到原来的工作流。

Kanbots 有点不一样。它直接把 Agent 塞进了看板卡片里。

这是 Hacker News 上一个开源项目（来源：Kanbots 官网，kanbots.dev，项目本身是线索级，star 数不多，但我看了它 GitHub 仓库的 README）。它的逻辑很简单：你在看板上创建一个任务卡片，比如“审查 PR #42 的代码量”，然后给这个卡片绑定一个 Agent 配置，Agent 就会自己去拉代码、跑 lint、生成审查报告，把结果贴回卡片。再比如，“部署 staging 环境”卡片，Agent 自动触发 CI/CD，然后把部署日志和状态回填。

我当时脑子里第一反应不是“牛X”，而是——这不就是我日常最烦的那些重复操作吗？

我在养猪设备厂一个人扛 IT，服务器、内网、ERP、开发需求，一锅端。我自己的待办清单是用一个本地 Markdown 文件记的，每天打开就是长长一串。很多时候不是事情难，是切换和跟踪太累——查完日志切回邮件，写完 SQL 又去重启服务。

如果每张卡片有 Agent，我只需要描述任务，它去执行然后汇报结果，我能省多少切换成本？

但紧接着我就开始犹豫了。这个工具刚起步，文档还很简陋（我翻 GitHub Wiki，只找到三个简单示例）。它依赖的 Agent 框架是它自己封装的，意味着如果出问题，我得去翻源码，而我的时间本来就不够。另外，让 Agent 有权限去操作代码仓库、服务器，安全边界怎么设？我自己写脚本都要小心测试，敢直接把服务器 SSH 密钥交给一个星星才两百多的开源项目吗？

坦率地讲，我大概率不会马上在生产环境用。但这件事让我停下来的点，不是工具本身，而是“Agent 嵌入工作流”这个范式开始变得很轻、很具体，不再是那种要折腾半天才能跑通的玩具。

我联想到上个月看到的一个帖子——Reddit r/Entrepreneur 上有人把同一个 AI 产品做成网站和 WhatsApp 机器人分别测试，结果 WhatsApp 那个转化率是网站的 3 倍多（他说是从 2.3% 到 7.8%，但这只是他个人的实验数据，没有第三方验证，算线索级）。这其实和 Kanbots 的思路相通：AI 能力不是单独造一个入口让用户去适应，而是直接融入到用户已经习惯的工具和流程里。看板本来就是开发者管任务的界面，WhatsApp 本来就是拉美和东南亚用户日常聊天的界面。把 Agent 塞进去，用户几乎不需要学习成本。

这让我的副业探索多了一层思考：如果我要做一个小产品，是做一个独立的网站，还是做成一个 Telegram 机器人、一条 WhatsApp 消息就能触发的东西？前者显得更“正规”，但后者可能更“有用”。而我这种人，没钱没时间做推广，可能更需要那种“触手可及”的渠道。

当然，Kanbots 本身离成熟还很远。我没实际用它跑过真实任务，只是本地用 Docker 起了 demo，创建了一张“生成今日服务器负载报告”的卡片，Agent 去调了 Grafana API，给我返回了一串 JSON，然后……卡住了。报错说模板渲染失败，我看了下日志是它默认的 Markdown 模板路径写死了，而我的环境路径不同。我手动改了源码里的路径，再试，成功了。整个过程花了我一个多小时。

这才是真实的 AI Agent 现状：概念性感，落地一堆小坑。

所以我现在的心态是：我会继续关注它，但暂时不会把它放进日常。也许下个月有空了，我会在自己的看板上挑几个低风险任务（比如自动生成每周工作摘要、检查 SSL 证书有效期）真正让它跑起来试两周。但我得承认，以我“研究到七八成然后停在门口”的惯性，这事也可能就搁置了。

不过至少，今天我又站在门口往里张望了一次。
