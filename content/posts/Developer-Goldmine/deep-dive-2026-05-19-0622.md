**标题**：两小时搭建Git作者标记防垃圾系统（零成本）  
**启动成本**：0元  
**时间投入**：2小时  
**预期收益**：每周省20小时手动审查，避免仓库被AI垃圾淹没

## 效率痛点
- **场景**：你维护一个GitHub开源项目，每天10+个PR和Issue是AI机器人写的，内容由大模型生成，无逻辑，占屏幕空间。手动一个一个标记“spam”并关闭，每天耗时30分钟。  
- **不用此方案的代价**：每月浪费15小时在甄别上（30分钟/天 × 30天 = 15小时）。如果这15小时用来接外包（按最低价50元/小时算），损失750元/月。

## 自动化方案
- **工具链**：GitHub + 自定义CI脚本（基于Git --author标志）。  
- **原理**：Git commit可以使用`--author`参数伪造作者。如果PR中的私有提交（来自仓库的合并提交）作者邮箱匹配不到GitHub账户活跃星标，就自动标记为可疑。配合GitHub的“Require prior contribution”设置，限制非贡献者直接合并。

## 从安装到运行的完整步骤
1. **启用仓库规则**：在GitHub仓库 → Settings → Branches → 保护分支规则 → “Require pull request reviews before merging” + “Do not allow bypassing the above settings”。  
2. **创建CI文件**：在仓库根目录新建 `.github/workflows/spam-check.yml`，内容如下：
```yaml
name: Spam Check by Author
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  check-author:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Get PR commit authors
        id: authors
        run: |
          # 获取当前PR中所有commits的作者邮箱
          BASE=$(jq -r '.pull_request.base.sha' "$GITHUB_EVENT_PATH")
          HEAD=$(jq -r '.pull_request.head.sha' "$GITHUB_EVENT_PATH")
          AUTHORS=$(git log --format='%ae' "$BASE".."$HEAD" | sort -u)
          echo "authors=$AUTHORS" >> $GITHUB_OUTPUT
      - name: Check if any author is known
        run: |
          if [[ -z "$(curl -s "https://api.github.com/users/bot-test-user" | jq -r '.login')" ]]; then
            echo "作者不存在，标记为spam"
            exit 1
          fi
```
   **注意**：上面示例中使用了虚假的用户查询。实际应用中应查询贡献者列表，或替换为你的白名单。  
3. **配置拒绝合并**：在上一步CI脚本`exit 1`时，PR的CI检查会失败，阻止合并。  
4. **测试**：用`git commit --author="Spam Bot <spam@example.com>"`创建一个测试提交，推送到新分支并提交PR，验证CI标记失败。

## 效果量化
- **优化前**：手动关闭15个垃圾PR/周 × 10分钟/个 = 150分钟/周 = 2.5小时/周 = 10小时/月。  
- **优化后**：CI自动标记，0人工干预。仅需每月检查一次误杀（约5分钟）。  
- **节省**：10小时/月，接外包按50元/小时 = 500元/月收益。

## 进阶技巧
1. **配合GitHub Action的注释**：CI失败后自动在PR中留言“此PR作者未被验证”，减少人工回复时间。  
2. **白名单机制**：使用仓库的CONTRIBUTORS文件作为白名单，通过`curl -s https://api.github.com/repos/OWNER/REPO/contributors | jq -r '.[].login'`实时拉取。  
3. **多维度评分**：结合提交时间、文件名模式（如只改README.md的PR标记为可疑），进一步提升准确率。

## 常见排障
1. **问题**：CI运行时报错 `jq not found`。  
   **解决**：在steps中添加`- name: Install jq` `run: sudo apt-get update && sudo apt-get install -y jq`。  
2. **问题**：误伤正常用户（如第一次贡献的新人）。  
   **解决**：将CI规则改为只标记不阻止，在CI输出中加上`exit 0`，但通过GitHub Issue标签标记PR为`need-verification`。  
3. **问题**：AI机器人绕过作者伪造（用真实用户的邮箱）。  
   **解决**：增加额外检查——要求PR提交者必须关联已存在issue或comment，否则标记。

## 硬要求说明
- 所有工具免费（GitHub + GitHub Actions免费额度足够）。  
- 代码可复制粘贴运行，依赖为bash + jq（标准Linux环境）。  
- 支持Windows/macOS？CI在云端执行，不依赖本地环境。本地测试可用Git Bash。  
- 如果GitHub Actions配额用尽：替代方案为用`pre-commit`钩子检查本地提交，或迁移到其他CI（如Jenkins免费版）。
