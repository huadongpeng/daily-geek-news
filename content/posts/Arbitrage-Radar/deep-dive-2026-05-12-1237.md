## 核心洞察：YouTube搜索太烂，几百人在抱怨，没人搞定

Reddit帖子"A video search engine that doesn't suck"里，用户说"there are only 5 or 6 relevant videos"之后就全是无关内容。这条抱怨被50多人顶帖，有人直接开骂"YouTube search is garbage"。

这不是技术难题——YouTube本身有API可以搜视频，但默认排序和推荐算法对深度搜索不友好。普通用户想要的是：搜"react tutorial 2026"的时候，能按时长、年代、语言、有没有字幕等条件精确匹配。

**别人为什么还没赚这个钱？** 不是技术壁垒，是信息差。大多数独立开发者忙着给AI套壳，忽略了这种"无聊但刚需"的方向。而你的优势：有Python基础，有免费API，能在家里完成。

**机会具体是什么？** 做一个垂直YouTube搜索引擎，只针对"教程"和"技术类"内容，提供YouTube原生没有的筛选维度（如：视频时长<10分钟、发布日期距今<30天、有英文字幕）。

## 执行清单Day1-7（每天≤30分钟）

**Day1（周三）：验证需求规模**
打开Reddit帖子链接（https://www.reddit.com/r/SideProject/comments/xxxx/），用Chrome的Inspect工具查看评论区，复制所有二级评论。用Python统计关键词："search"、"find"、"filter"、"sucks"。如果出现频次>10次，就说明需求真实。

**Day2（周四）：注册YouTube API**
打开console.cloud.google.com，创建一个新项目，启用YouTube Data API v3。创建OAuth 2.0凭据（桌面应用类型），下载JSON密钥。这一步免费，但需要Google账号。用这个Python脚本测试是否能搜到视频：
```python
from googleapiclient.discovery import build
import json

api_key = "YOUR_API_KEY" # 在凭据页面创建API key
youtube = build('youtube', 'v3', developerKey=api_key)
request = youtube.search().list(
    q='react tutorial 2026',
    part='snippet',
    maxResults=10,
    type='video',
    order='viewCount'
)
response = request.execute()
for item in response['items']:
    print(item['snippet']['title'], item['id']['videoId'])
```

**Day3（周五）：添加筛选条件**
修改脚本，加入"videoDuration"参数（long/medium/short），筛选5-15分钟长度的教程。再加"publishedBefore"和"publishedAfter"，筛出近30天的视频。输出结果为CSV文件：标题、链接、时长、发布日期。

**Day4（周六）：做展示页面**
用Flask或FastAPI写个简单的Web界面，只有一个搜索框。用户输入关键词后，后端调用API返回结果。部署到Render（免费版）。域名可以不买，用render的免费子域名。

**Day5（周日）：发布到V2EX和Reddit**
在V2EX的"分享创造"板块发帖，标题"做了个YouTube教程搜索工具，能按时长日期筛选"。同时去Reddit原帖下回复"I built a simple demo for this"加链接。目标：收集10个用户反馈。

**Day6（周一）：收集需求迭代**
分析用户反馈，重点关注"还缺什么"。常见需求可能是：按频道筛选、按语言筛选、支持下载字幕。把最热门的需求加入下一个版本。

**Day7（周二）：加入变现尝试**
在页面底部加一个"赞助"按钮，通过Patreon或Buy Me a Coffee链接（免费注册）。不强制收费，但告诉用户"如果你觉得有用，请请我喝咖啡"。同时发到Product Hunt（免费）做暴露。

## 3个最容易踩的坑

**坑1：过度优化功能才上线**
错误做法：做成了"完美版"才发布，但发现用户根本不来。正确做法：Day5必须上线最小原型（只有搜索+显示结果），上线后再根据反馈加功能。每多一天不上线，就少一天知道真实需求。

**坑2：依赖API配额不够用**
YouTube API免费配额每天10000单元，每次搜索消耗100单元。如果你做了"无限滚动"，用户每次滚动都是新搜索。正确做法：前端缓存搜索结果，同一关键词重复搜索不调用API；或者限制免费用户每天搜5次，可付费解锁更多次数。

**坑3：卖功能而不是卖解决方案**
如果只是说"我的搜索器能筛选时长"，用户不会付钱。要说"帮你5分钟内找到2026年最新讲解视频，不比每天刷YouTube浪费时间"。在页面文案里突出"时间节省"，而不是"功能列表"。
