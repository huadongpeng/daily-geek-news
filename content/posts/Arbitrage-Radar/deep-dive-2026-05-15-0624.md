# 核心洞察

2026年5月14日，安全研究员Zhenpeng Lin公开了Nginx中存在18年的远程代码执行漏洞（CVE-2026-42945）。这个漏洞存在于ngx_http_rewrite_module模块中，允许未认证攻击者远程执行代码。Reddit/webdev上大量运维人员抱怨：“又有一堆服务器要熬夜升级了”。更有人骂：“Nginx这么多年还没修好底层问题”。

这个钱为什么别人没赚到？不是技术壁垒，是信息差。很多小公司和个人站长根本没有订阅安全通知，也不会主动查CVE。你能做的不是卖修复服务，而是帮助别人检测和修复——尤其是那些不懂命令行的人。

# 执行清单 Day1-7

## Day1（周一）：确定自己是否受影响
1. 打开终端→输入 `nginx -v` 查看版本号
2. 如果版本低于 1.26.3 或 1.27.4，就受影响
3. 打开 https://nginx.org/en/download.html → 确认最新稳定版是 1.26.3
4. 记录下当前版本和操作系统

## Day2（周二）：备份配置文件
1. 输入 `cp -r /etc/nginx /etc/nginx_backup`
2. 输入 `nginx -t` 确认配置无误
3. 如果报错，记下错误信息

## Day3（周三）：升级Nginx（Debian/Ubuntu）
1. 输入 `sudo apt update`
2. 输入 `sudo apt upgrade nginx`
3. 输入 `nginx -v` 确认版本
4. 输入 `systemctl restart nginx`

## Day4（周四）：升级Nginx（CentOS/RHEL）
1. 输入 `sudo yum update nginx` 或 `sudo dnf update nginx`
2. 输入 `nginx -v` 确认版本
3. 输入 `systemctl restart nginx`

## Day5（周五）：检查是否被入侵
1. 查看 `/var/log/nginx/access.log` 中是否有异常请求
2. 搜索关键词 `rewrite` 和可疑IP
3. 运行 `last` 查看登录记录

## Day6（周六）：加固安全
1. 编辑 `/etc/nginx/nginx.conf`，添加 `server_tokens off;` 隐藏版本号
2. 安装fail2ban：`sudo apt install fail2ban`
3. 配置fail2ban保护Nginx

## Day7（周日）：自动化监控
1. 设置cron任务每周检查更新：`0 2 * * 0 apt-get update -y && apt-get upgrade -y`
2. 开启系统自动安全更新

# 3个最容易踩的坑

1. **不备份直接升级**：新版本可能不兼容旧配置。避开：升级前必须 `cp -r /etc/nginx /etc/nginx_backup`。
2. **忘记重启服务**：升级完不重启等于没修。避开：升级后立刻执行 `systemctl restart nginx`。
3. **忽略负载均衡器**：如果你用Nginx做反向代理，重启会造成短暂断连。避开：安排在凌晨低峰期操作。
